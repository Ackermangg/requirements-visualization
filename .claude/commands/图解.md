# 需求图解生成器

**目标读者**：刚拿到需求的开发人员。他们能看懂技术，但不了解业务背景。需要在 1 分钟内建立"业务上下文"——理解这个需求**要做什么**，以及计算逻辑/规则**为什么这样设计**。

图解的本质是：**把抽象的规则和计算参数，放回它们存在的真实业务场景里**，让开发人员一眼看懂"哦，原来是这么回事"。

---

## 第一步：提炼四个核心信息（先输出给用户确认）

**① 业务场景（一句话）**
- 谁、在什么情况下、遇到了什么问题？
- 格式：「[角色] 在 [场景] 时，[痛点]，所以系统要做 [什么]」

**② 功能本质**
- 系统在背后做了什么事？（用户视角，不是实现视角）
- 输入是什么，输出是什么，对用户产生了什么变化？

**③ 计算规则/参数/阈值背后的业务理由（核心！）**

这是最有价值的部分。需求里往往充满数字和条件（19T、100km、5h、>9.6m……），开发人员看不懂它们为什么是这个值。你需要挖掘每一个关键参数/规则的**业务原因**：

- 这个数字/条件对应现实中的什么物理约束或业务规则？
- 如果没有这条规则，会发生什么不合理的事？
- 这个参数为什么是这个值而不是更大或更小？

**分析方法**：针对需求中每个关键判断条件，问自己：
> "这条规则在现实业务中保护的是什么？排除的是什么情况？"

例如：
- `线路货量 < 19T × 要车数` → 19T 是省际大车满载上限，低于此说明有余载，才值得绕道接货；超过则车已满，绕道亏本
- `出港分拨距离 < 100km` → 超过100km绕道成本超过收益，100km是经济合理的"顺路"边界
- `当前时间 - 发车时间 ≤ 5h` → 保证调度员有足够时间重新安排，太晚了车已经要开走
- `出港为点部/二级，进港为一级` → 小站发往大枢纽是货量天然不足的典型场景，最容易有余载需要补货

识别 3-5 个这样的规则，每个写清楚：**规则 + 它在现实中保护/排除了什么**

**④ 开发注意事项**
- 哪些参数需要做成可配置项？
- 有哪些双向操作（打标/清标）或边界情况？
- 最多 3 条，简洁直接

## 第二步：规划图组结构（分镜策略）

**默认 1 张图，最多 4 张图**。先想清楚整组图的叙事结构，再逐一生成。

### 张数决策

| 需求复杂度 | 推荐张数 |
|-----------|---------|
| 简单改动（加字段/调阈值/小逻辑调整） | **1 张**，4:3 比例 |
| 中等（有明确场景 + 3-5 条核心规则） | **1 张**，16:9 |
| 复杂（多个维度，单图需标注 5+ 规则或 4+ 实体） | **2-3 张** |
| 很复杂（触发→筛选→计算→结果各有重要逻辑） | **3-4 张** |

**拆分判断**：如果单张图里需要塞入超过 5 个规则标注，或场景实体超过 4 个，就应该拆分——信息过密的图比多张清晰的图更难读。

### 图组分镜模板（复杂需求参考）

多图时，每张图是故事的一个"章节"，按以下顺序规划（不必全用，按实际需要取舍）：

```
图1：触发场景 — 为什么要做这个功能？
     谁在什么情况下遇到了什么问题，系统何时介入
     重点：展示业务痛点和触发时机，建立背景认知

图2：筛选逻辑 — 系统先排除哪些不合适的情况？
     哪类对象/线路/任务才会被纳入处理，哪类直接跳过
     重点：每个筛选条件标注在它约束的实体旁，解释为什么有这条门槛

图3：核心计算/匹配规则 — 系统怎么评估/打分/匹配？
     核心算法的业务含义：参数代表什么，阈值为什么是这个值
     重点：把计算逻辑放回真实场景，让数字有现实依据

图4：结果与操作 — 系统最终做了什么，用户看到什么？
     打标/推荐/通知的形态，调度员/用户的操作界面示意
     重点：展示"输入→系统决策→输出"的最终状态
```

**原则**：
- 每张图独立可读，不依赖前一张才能看懂
- 图之间有逻辑递进（从"为什么"到"怎么做"到"结果是什么"）
- 不强求凑满 4 张，只在内容真正无法压缩时才拆分

在开始生成前，**先输出本次的图组规划**（每张图的章节标题 + 一句话主题），供确认后再逐一生成。

## 第三步：构建场景示意图（核心策略）

**这是关键策略**：图解的主要形式是**业务场景示意图**，而不是信息卡片布局。

### 什么是场景示意图？

用一幅画面展示这个需求所处理的**真实业务情境**：有地点、有角色、有动作、有结果。规则和参数标注在场景中对应的位置上，让人一眼理解"这条规则在保护/排除什么"。

- 货运路线需求 → 画地图/路线图，标注城市、货车、货量、距离
- 调度决策需求 → 画调度员的工作界面场景，标注触发条件
- 数据计算需求 → 画数据流向或对比图，把数字放在它对应的实体上
- 状态流转需求 → 画对象在不同状态下的样子，而不是状态机图

### 场景图的三层结构

```
┌─────────────────────────────────────────────────┐
│  标题：[需求一句话本质]                            │
│  副标题：[场景定位]                               │
├─────────────────────────────────────────────────┤
│                                                 │
│           核心场景插图                            │
│  （地图/界面/实体示意图，含角色、动作、数据）       │
│                                                 │
│  规则直接标注在场景中对应位置：                    │
│  "为什么X？" → 标注在场景里的X旁边                │
│                                                 │
├─────────────────────────────────────────────────┤
│  底部注意事项：[可配置项] [清标条件] [性能提示]    │
└─────────────────────────────────────────────────┘
```

### 构建 Prompt 的方法

**第一：描述场景的画面感**（像写分镜脚本一样，不要像写文档）

```
Create a stylized [地图/界面/流程] diagram illustrating [场景一句话描述].
This is a SCENARIO ILLUSTRATION — show the real business situation visually.
NO text card layouts, NO flowchart boxes, NO numbered steps.

BACKGROUND: [描述背景风格：如 light blue-gray map background / light dashboard style]

[实体A] — positioned [left/right/top], labeled '[标签]' in Chinese:
- [描述这个实体的视觉状态：如 a truck shown 35% loaded / a warehouse icon / a person looking at screen]
- Annotation badge: '[规则] — [用一句话解释为什么]' in Chinese

[实体B] — positioned [center/right], labeled '[标签]' in Chinese:
- [描述视觉状态]
- [相关规则标注]

[动作/关系线]:
- [原始路径]: dashed gray [方向] arrow, labeled '[描述]' in Chinese
- [推荐/新路径]: bold orange [方向] arrow, labeled '[描述]' in Chinese

[范围/边界标记]:
- [如距离圈、时间窗口标记]: [描述]，label '[规则+原因]' in Chinese

BOTTOM STRIP (dark navy):
'[开发注意：可配置项、清标条件等]' in white Chinese text
```

**第二：规则标注的写法**

规则要标注在场景中**规则所约束的那个实体旁边**，格式：
```
Annotation near [实体]: '[规则]' bold + '[一句话业务原因]' smaller text, both Chinese
```

例如：
- 货车旁：`'货量 < 19T×要车数' — 省际大车满载约19T，低于此才有余载值得绕道`
- 距离圈边：`'< 100km' — 超过此距离绕道成本超过收益`
- 时钟图标旁：`'发车≤5h内' — 保证调度员有足够时间重新安排`

**第三：视觉风格指引**

```
Visual style: Clean logistics map illustration / flat design dashboard style.
Colors:
- Dark navy (#1E3A5F): titles, major labels
- Orange (#F59E0B) or green (#10B981): recommended path / positive outcome
- Gray (#6B7280): original/baseline state
- Light backgrounds for annotations: yellow / green / orange tinted boxes
Chinese typography throughout. Flat icon style (buildings, trucks, people).
The diagram should make developers immediately understand WHY each rule exists
by seeing it in the context of the real business scenario.
```

**第四：宽高比选择**
- 横向场景（A在左、B在右、C在上方）→ `16:9`
- 纵向场景或信息较多 → `4:3`

## 第四步：生成图片

逐张调用 `generate_image` 工具，**每张生成完毕后立即输出路径和说明，不要等全部生成完再统一输出**：

- **prompt**: 第三步构建的完整 prompt
- **outputPath**: `D:/dev/RequirementsVisualization/output/infographics/{需求简短名称}_{序号}_{时间戳}.png`
  - 单张图不加序号；多张图加序号后缀：`_01`、`_02`……
  - 时间戳格式: YYYYMMDD_HHmmss
- **aspectRatio**: 根据第三步判断选择

若某张生成失败（无图像数据返回），说明 prompt 太复杂，简化后重试：移除部分标注，保留最核心的 2-3 个规则解释。单张失败不影响其他张的生成。

---

## 第五步：输出结果

1. **📋 需求解读**（第一步的四部分，通俗中文，不超过 400 字）
2. **🖼️ 图组说明**：每张图的路径 + 一句话说明（这张图展示什么章节、各区域对应什么）：
   ```
   图1 [触发场景] → output/infographics/xxx_01_....png
        展示：调度员发现线路货量不足时的决策场景
   图2 [筛选逻辑] → output/infographics/xxx_02_....png
        展示：哪些线路/要车任务会进入经停推荐候选池
   ```

**用户反馈处理：**
- "规则解释不对" → 重新理解该规则的业务含义，修正标注后重新生成
- "看起来还是像流程图" → 去掉步骤箭头，改用场景实体（人/车/货/地点）来承载规则
- "文字太多/看不清" → 减少标注数量，只保留最关键的 2-3 个规则解释
- "场景画错了" → 先输出场景描述让用户确认，再生成图片

---

## 特殊场景处理

**计算型需求（满是公式和条件）**：
- 先找"这些计算最终是为了判断什么"，画出判断结果对应的业务场景
- 规则标注重点放在：这个条件排除了哪类不合适的情况

**改进型需求（在现有功能上调整）**：
- 用"旧场景 vs 新场景"的对比画面，重点展示哪里不一样以及为什么改

**纯界面型需求（加字段/改交互）**：
- 画简化的界面 mockup，标注每个设计决策的业务理由
- 重点放在"为什么放在这个位置/时机"

**截图输入**：先用自己的话复述截图内容，确认理解正确，再进行分析和生成。

**需求很长（>2000字）**：忽略实现细节，聚焦场景本质，最多提炼 4 个核心规则进行标注。
