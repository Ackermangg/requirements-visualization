# 需求图解生成器

**目标读者**：刚拿到需求的开发人员。他们能看懂技术，但不了解业务背景。需要在 1 分钟内建立"业务上下文"——理解这个需求**要做什么**，以及计算逻辑/规则**为什么这样设计**。

图解的本质是：**把抽象的规则和计算参数，放回它们存在的真实业务场景里**，让开发人员一眼看懂"哦，原来是这么回事"。

---

## 第一步：提炼四个核心信息（先输出给用户确认）

**① 业务场景（一句话）**
- 谁、在什么情况下、遇到了什么问题？
- 格式：「[角色] 在 [场景] 时，[痛点]，所以系统要做 [什么]」

**② 功能本质**
- 系统在背后做了什么事？（用户视角，不是实现视角）
- 输入是什么，输出是什么，对用户产生了什么变化？

**③ 计算规则/参数/阈值背后的业务理由（核心！）**

这是最有价值的部分。需求里往往充满数字和条件，开发人员看不懂它们为什么是这个值。你需要挖掘每一个关键参数/规则的**业务原因**：

- 这个数字/条件对应现实中的什么物理约束或业务规则？
- 如果没有这条规则，会发生什么不合理的事？
- 这个参数为什么是这个值而不是更大或更小？

**分析方法**：针对需求中每个关键判断条件，问自己：
> "这条规则在现实业务中保护的是什么？排除的是什么情况？"

识别 3-5 个这样的规则，每个写清楚：**规则 + 它在现实中保护/排除了什么**

**④ 开发注意事项**
- 哪些参数需要做成可配置项？
- 有哪些双向操作（打标/清标）或边界情况？
- 最多 3 条，简洁直接

---

## 第二步：规划图组结构（分镜策略）

**默认 1 张图，最多 4 张图**。先想清楚整组图的叙事结构，再逐一生成。

### 张数决策

| 需求复杂度 | 推荐张数 |
|-----------|---------|
| 简单改动（加字段/调阈值/小逻辑调整） | **1 张**，4:3 比例 |
| 中等（有明确场景 + 3-5 条核心规则） | **1 张**，16:9 |
| 复杂（多个维度，单图需标注 5+ 规则或 4+ 实体） | **2-3 张** |
| 很复杂（触发→筛选→计算→结果各有重要逻辑） | **3-4 张** |

**拆分判断**：如果单张图里需要塞入超过 5 个规则标注，或场景实体超过 4 个，就应该拆分。

### 图组分镜模板（复杂需求参考）

```
图1：触发场景 — 为什么要做这个功能？
图2：筛选逻辑 — 系统先排除哪些不合适的情况？
图3：核心计算/匹配规则 — 系统怎么评估/打分/匹配？
图4：结果与操作 — 系统最终做了什么，用户看到什么？
```

**原则**：每张图独立可读；图之间有逻辑递进；不强求凑满 4 张。

在开始生成前，**先输出本次的图组规划**（每张图的章节标题 + 一句话主题），供确认后再逐一生成。

---

## 第三步：构建场景示意图（核心策略）

**关键**：图解的主要形式是**业务场景示意图**，而不是信息卡片布局或流程图。

用一幅画面展示这个需求所处理的**真实业务情境**：有地点、有角色、有动作、有结果。规则和参数标注在场景中对应的位置上。

- 货运路线需求 → 画地图/路线图，标注城市、货车、货量、距离
- 调度决策需求 → 画调度员的工作界面场景，标注触发条件
- 数据计算需求 → 画数据流向或对比图，把数字放在它对应的实体上
- 状态流转需求 → 画对象在不同状态下的样子，而不是状态机图

### Prompt 构建方法

**描述场景的画面感**（像写分镜脚本，不要像写文档）：

```
Create a stylized [地图/界面/流程] diagram illustrating [场景一句话描述].
This is a SCENARIO ILLUSTRATION — show the real business situation visually.
NO text card layouts, NO flowchart boxes, NO numbered steps.

BACKGROUND: [描述背景风格]

[实体A] — positioned [left/right/top], labeled '[标签]' in Chinese:
- [描述视觉状态]
- Annotation badge: '[规则] — [一句话业务原因]' in Chinese

[动作/关系线]:
- [原始路径]: dashed gray arrow, labeled '[描述]' in Chinese
- [推荐路径]: bold orange arrow, labeled '[描述]' in Chinese

BOTTOM STRIP (dark navy):
'[开发注意：可配置项、清标条件等]' in white Chinese text
```

**规则标注写法**：标注在场景中**规则所约束的那个实体旁边**：
```
Annotation near [实体]: '[规则]' bold + '[一句话业务原因]' smaller text, both Chinese
```

**视觉风格**：
```
Visual style: Clean logistics map illustration / flat design dashboard style.
Colors:
- Dark navy (#1E3A5F): titles, major labels
- Orange (#F59E0B) or green (#10B981): recommended path / positive outcome
- Gray (#6B7280): original/baseline state
- Light backgrounds for annotations: yellow / green / orange tinted boxes
Chinese typography throughout. Flat icon style (buildings, trucks, people).
```

**宽高比**：横向场景 → `16:9`；纵向或信息密集 → `4:3`

---

## 第四步：生成图片

逐张调用 `generate_image` 工具，**每张生成完毕后立即输出路径和说明**：

- **outputPath**: `{PROJECT_DIR}/output/infographics/{需求简短名称}_{序号}_{时间戳}.png`
  - 单张图不加序号；多张图加序号后缀：`_01`、`_02`……
  - 时间戳格式: YYYYMMDD_HHmmss
- **aspectRatio**: 根据第三步判断选择

若某张生成失败（无图像数据返回），简化 prompt 后重试：移除部分标注，保留最核心的 2-3 个规则解释。

---

## 第五步：输出结果

1. **📋 需求解读**（第一步的四部分，通俗中文，不超过 400 字）
2. **🖼️ 图组说明**：每张图的路径 + 一句话说明

**用户反馈处理：**
- "规则解释不对" → 重新理解该规则业务含义，修正后重新生成
- "看起来还是像流程图" → 去掉步骤箭头，改用场景实体承载规则
- "文字太多/看不清" → 减少标注，只保留最关键的 2-3 个规则
- "场景画错了" → 先输出场景描述让用户确认，再生成

---

## 特殊场景处理

**计算型需求**：先找"这些计算最终判断什么"，画判断结果对应的业务场景，规则标注重点放在"这个条件排除了哪类情况"。

**改进型需求**：用"旧场景 vs 新场景"对比画面，重点展示哪里不一样及为什么改。

**纯界面型需求**：画简化的界面 mockup，标注每个设计决策的业务理由。

**截图输入**：先用自己的话复述截图内容，确认理解正确，再分析生成。

**需求很长（>2000字）**：忽略实现细节，聚焦场景本质，最多提炼 4 个核心规则。
